Overview / Live proof of concept

Live decisioning on real shop-floor data, actually built and running

Not a slide of logos — a working system. Two independent OT sources — Nissan Smyrna's own Cimplicity SCADA and an Ignition stand-in for the non-Cimplicity plant areas — publish over OPC UA into a single, standards-governed Unified Namespace. HighByte Intelligence Hub runs that namespace; a second, independent vendor (MaestroHub) reads the exact same feed with zero translation; and Humble's Eva reads it to reason over causal chains, rank interventions, and answer questions against the live database — the actual maintenance-cockpit value. The point of the two-source, two-reader shape: neither the source you have nor the platform you pick ever gates the value.

Scope — read this first. The architecture pattern is proven live, end to end: Ignition → HighByte → UNS is publishing right now (verified in HighByte's own UNS Client and mirrored by MaestroHub), and Eva reasons over the feed. We are now bringing Nissan Smyrna's real Cell 056 Cimplicity onto the identical spine — the asset model is already generated from Nissan's own Cimplicity export and loaded in HighByte. Telemetry values shown are representative/simulated until source tags are bound. The further-along, cell-022-specific build is documented on the Fascia & Plastics track.
The stack we built

Two independent OT sources, one governed namespace, any consumer

The Unified Namespace — an i3X / CESMII SM-Profiles-governed MQTT topic tree — is the spine. Sources publish into it; consumers read from it. Neither side is coupled to the other, which is exactly why the platform question never has to stall the value question.

3 — Not locked to one platform, or one source

MaestroHub reads the same feed; Ignition proves any source rides the same pipe

MaestroHub — a completely separate product from a different vendor, and the platform this namespace originally ran on — subscribes to HighByte's broker as a read-only client: same topics, same payloads, zero translation. And the Ignition leg proves the mirror image on the source side: a non-Cimplicity system publishes onto the identical UNS with the identical pattern. Whatever platform Nissan lands on, and whatever mix of source systems the plant runs, the namespace keeps working.

What this proves

Real decisioning value, on a namespace locked to neither a vendor nor a source

  • Two independent OT sources — Nissan's own Cimplicity and an Ignition stand-in — publish onto one governed UNS with the same pattern, so non-Cimplicity plant areas are first-class, not an exception.
  • Equipment is modeled against real published standards (OPC UA companion specs, ISO 5393, PackML), not ad-hoc tags — so the decisioning layer above is trustworthy.
  • A second, independent vendor platform consumes the same feed with zero translation — the platform decision is decoupled from the value already flowing.
  • The asset model is generated from Nissan's own Cimplicity exports — the structure Eva reasons over is the structure the plant floor actually uses.
  • Eva turns that feed into causal chains, ranked decisions, and answerable questions — and the Cell 056 Asset Health and OEE cockpit is built on the same topics, in whatever tool Nissan prefers.

What's still ahead before this is Nissan's real system

  • Bind the model attributes to live source tags instead of representative values, once source access is confirmed. Detail on the Fascia & Plastics track.
  • Security review of the MQTT broker's external-subscriber credential model before anything touches production OT data.
  • Decide the platform of record on Nissan's own timeline — this page's whole point is that decision no longer has to gate operator and maintenance value.
