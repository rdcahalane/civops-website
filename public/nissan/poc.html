Overview / Live proof of concept

The UNS / i3X architecture, actually built and running

Not a slide of logos — a working pipeline. Five shop-floor equipment types publish live telemetry into a Unified Namespace, a schema registry enforces the standards contract on every message, a dashboard renders it live, and a second, independent vendor platform subscribes to the same feed with zero data translation. This page is the screenshots from that running system.

Scope — read this first. This build validates the architecture pattern (i3X-style Unified Namespace, standards-grounded schemas, cross-vendor interoperability) using representative equipment and simulated telemetry — it is not connected to Nissan Smyrna's real Cimplicity data. The real, cell-022-specific build — generated directly from Nissan's own Cimplicity export (ABB sealer robots, actual point addresses) — is a separate, further-along workstream documented on the Fascia & Plastics track. This page answers "does the pattern work end to end"; that one answers "is it wired to your actual paint shop."

1 — Orchestration

Webhook in, standards-modeled payload out

Each pipeline is a visual, three-node flow: a webhook trigger accepts the raw equipment event, a transform stage maps it onto the standards-grounded field set, and a publish stage writes it to the equipment's UNS topic. No custom code — the whole thing is built and versioned in the designer.

2 — The contract

Schemas enforce the standard, not just document it

Every UNS topic is bound to a versioned schema. The schema isn't paperwork — the platform validates every published message against it, and the description field cites exactly which standard grounds each attribute, so the "why this field" question is answered in the tool, not a separate spec doc.

UNS Schema Registry

Left rail shows all five schemas grouped by kind (Assembly State, Conveyor State, Press State, Robot State ×2). Each attribute lists its type, cardinality, and a description that names the source standard — visible, auditable, not tribal knowledge.

4 — The interoperability proof

A second, independent vendor platform reads the same data

This is the part that actually matters for a technology decision: HighByte Intelligence Hub — a completely separate product from a different vendor — was pointed at the same MQTT-based Unified Namespace and subscribed to all five equipment topics with no data translation, no re-mapping, no custom connector work. It just read the standard.

HighByte Intelligence Hub — Connections → maestrohub_uns → Inputs
All 5 equipment topics, live, subscribed from a different vendor's platform
HighByte's own topic browser walked the real MQTT tree (mHv1.0 / axiom / nissan-ccr) and found all five topics on the first pass — proof this isn't a point-to-point integration, it's a genuine open namespace any standards-compliant subscriber can read.
Why this is the argument, not the demo: the moment two independently-built platforms can read the same shop-floor data with zero translation work, technology selection stops being a lock-in decision. Nissan keeps optionality — MaestroHub, HighByte, or a future replacement can all sit on top of the same standards-modeled namespace without re-integrating the plant floor.
What this proves

The architecture pattern works end to end

  • Equipment data can be modeled against real published standards (OPC UA companion specs, ISO 5393, PackML), not ad-hoc tags.
  • An orchestration layer can enforce that model at publish time via a schema registry — governance is in the platform, not a wiki page.
  • A live dashboard can bind directly to the namespace with no separate reporting/ETL layer.
  • A second, independent vendor platform can consume the same data with zero translation — the core i3X / UNS promise, demonstrated, not asserted.

What's still ahead before this is Nissan's real system

  • Point this same pattern at real Cimplicity tags for cell 022 — the groundwork is already done in the Fascia & Plastics track.
  • Replace the simulators with the actual OT/historian connections once source access is confirmed.
  • Security review of the MQTT broker's external-subscriber credential model before anything touches production OT data.
  • Decide MaestroHub vs. HighByte vs. another platform as the system of record — this page's whole point is that decision is now lower-risk either way.
