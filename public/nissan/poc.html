Overview / Live proof of concept

Live decisioning on real shop-floor data, actually built and running

Not a slide of logos — a working system. Five shop-floor equipment types publish live telemetry into a Unified Namespace; Humble's Eva reads it and reasons over causal chains, ranks intervention options, and answers questions against the live database — the actual maintenance-cockpit value. Underneath, the namespace is standards-governed and platform-neutral: HighByte Intelligence Hub runs the namespace today, and a second, independent vendor (MaestroHub) reads the exact same feed with zero translation, so the platform question never has to gate the value question.

Scope — read this first. This build validates the architecture pattern (i3X-style Unified Namespace, standards-grounded schemas, cross-vendor interoperability) using representative equipment and simulated telemetry — it is not connected to Nissan Smyrna's real Cimplicity data. The real, cell-022-specific build — generated directly from Nissan's own Cimplicity export (ABB sealer robots, actual point addresses) — is a separate, further-along workstream documented on the Fascia & Plastics track. This page answers "does the pattern work end to end, and does it produce a maintenance team can actually use"; that one answers "is it wired to your actual paint shop."

2 — The plumbing underneath

HighByte Intelligence Hub publishes the namespace

Five equipment-type pipelines — Webhook Trigger → Validate → Publish — take real telemetry and write it to HighByte's own embedded MQTT broker under the governed mHv1.0/axiom/nissan-ccr topic structure. HighByte's own UNS Client reads the same broker it just wrote to: no custom connector, no re-mapping.

3 — The same feed, read by a second vendor

MaestroHub reads it too — but nothing here is locked to one platform

MaestroHub — a completely separate product from a different vendor, and the platform this namespace originally ran on — now subscribes to HighByte's broker as a read-only client. Same topics, same payloads, zero translation. Whatever platform Nissan lands on for governance, the underlying UNS keeps working — so "data governance" debates don't have to stall the value operators and maintenance teams are already getting from Eva.

What this proves

Real decisioning value, on a namespace that isn't locked to one vendor

  • Eva can turn raw equipment telemetry into causal chains, ranked decisions, and answerable questions — the thing a maintenance lead actually uses, not a chart.
  • Equipment data can be modeled against real published standards (OPC UA companion specs, ISO 5393, PackML), not ad-hoc tags — so that decisioning layer is trustworthy.
  • A second, independent vendor platform can consume the same data with zero translation — the platform decision is decoupled from the value Nissan is already getting.
  • Net effect: ops and maintenance teams don't have to wait on a "data governance" decision to start getting value — the namespace works today, under whichever platform Nissan lands on.

What's still ahead before this is Nissan's real system

  • Point this same pattern at real Cimplicity tags for cell 022 — the groundwork is already done in the Fascia & Plastics track.
  • Replace the simulators with the actual OT/historian connections once source access is confirmed.
  • Security review of the MQTT broker's external-subscriber credential model before anything touches production OT data.
  • Decide the platform of record on Nissan's own timeline — this page's whole point is that decision no longer has to gate operator and maintenance value.
