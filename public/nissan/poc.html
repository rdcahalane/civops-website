Overview / Live proof of concept

Live decisioning on real shop-floor data, actually built and running

Not a slide of logos — a working system. Two independent OT sources — Nissan Smyrna's own Cimplicity SCADA and an Ignition stand-in for the non-Cimplicity plant areas — publish over OPC UA into a single, standards-governed Unified Namespace. HighByte Intelligence Hub runs that namespace; a second, independent vendor (MaestroHub) reads the exact same feed with zero translation; and Humble's Eva reads it to reason over causal chains, rank interventions, and answer questions against the live database — the actual maintenance-cockpit value. The point of the two-source, two-reader shape: neither the source you have nor the platform you pick ever gates the value.

Scope — read this first. The architecture pattern is proven live, end to end: Ignition → HighByte → UNS is publishing (verified in HighByte's own UNS Client and mirrored by MaestroHub), and Eva reasons over the feed. The web cockpit, the dashboards below, and the Ignition Perspective twin all run against the same namespace structure. We are bringing Nissan Smyrna's real Cell 056 Cimplicity onto the identical spine — the asset model is already generated from Nissan's own Cimplicity export and loaded in HighByte. Where a source tag isn't bound yet (e.g. the paused BC5602 torque leg, or a scrap/quality feed), values are clearly labeled representative. The further-along, cell-022-specific build is documented on the Fascia & Plastics track.
The stack we built

Two independent OT sources, one governed namespace, any consumer

The Unified Namespace — an i3X / CESMII SM-Profiles-governed MQTT topic tree — is the spine. Sources publish into it; consumers read from it. Neither side is coupled to the other, which is exactly why the platform question never has to stall the value question.

Standards, honestly

How i3X-compliant is each schema, really?

"i3X-compliant" should mean a schema conforms to a published, machine-readable information model — an OPC UA Companion Specification or a CESMII SM Profile — not just that it's named after a standard. By that bar, four of the five now sit on real companion specs (the torque station was re-grounded from ISO 5393 onto the OPC UA Tightening Systems companion spec — see below), and one is base-compliant with vendor-defined extensions. Stated plainly: none is a certified SM Profile yet — certification means registering and validating the type model, a defined next step, not a repaint.

The clean fix, applied: the torque station is now grounded on OPC 40451-1 — OPC UA for Tightening Systems (part of the VDMA-led OPC 40450 Joining Systems family, an actual information model with a result / torque / angle / evaluation model) instead of ISO 5393. Its live payload — torque_achieved_nm, target_torque_nm, angle_achieved_deg, pass_fail_result — maps directly onto that spec, putting it on the same compliance track as the robots and machinery. Everywhere else, the honest label remains "standards-grounded, SM-Profile-track," not "certified."
Worth knowing — Toyota is reportedly running a very similar UNS / information-model exercise, and may be open to collaborating or sharing that work. A cross-OEM push on common OPC UA companion-spec profiles would let Nissan adopt proven, shared models rather than define them alone — turning "standards-grounded" into "industry-aligned."

5 — Off the plant floor, onto the internet

A live UNS bridge — and the same screen in a second tool

The namespace no longer lives only on the plant network. A small read-only bridge subscribes to HighByte's broker and streams its real topics over a Cloudflare tunnel, so any authorized consumer — a browser, a BI tool, a second HMI platform — can read the live namespace without touching OT. What this proves

Real decisioning value, on a namespace locked to neither a vendor nor a source

  • Two independent OT sources — Nissan's own Cimplicity and an Ignition stand-in — publish onto one governed UNS with the same pattern, so non-Cimplicity plant areas are first-class, not an exception.
  • Equipment is modeled against real published standards (OPC UA companion specs, ISO 5393, PackML), not ad-hoc tags — so the decisioning layer above is trustworthy.
  • A second, independent vendor platform consumes the same feed with zero translation — the platform decision is decoupled from the value already flowing.
  • The asset model is generated from Nissan's own Cimplicity exports — the structure Eva reasons over is the structure the plant floor actually uses.
  • Eva turns that feed into causal chains, ranked decisions, and answerable questions — and the Cell 056 Asset Health and OEE cockpit is built on the same topics, in whatever tool Nissan prefers.

What's still ahead before this is Nissan's real system

  • Bind the model attributes to live source tags instead of representative values, once source access is confirmed. Detail on the Fascia & Plastics track.
  • Security review of the MQTT broker's external-subscriber credential model before anything touches production OT data.
  • Decide the platform of record on Nissan's own timeline — this page's whole point is that decision no longer has to gate operator and maintenance value.
