Architecture

From the systems in place today, through the gaps that block scale, to the recommended future state — and the detailed build and data-standardization work that follows.

Architecture story

Same base graphic, progressively built for clarity. Detailed architecture and data standardization tabs mark follow-on design work.

As-Is: the plant already has useful systems

Nissan does not need a "rip and replace" story. The current stack already contains valuable signals and work processes. The challenge is not lack of systems; it is fragmentation.

  • CIMPLICITY remains the live control and HMI backbone.
  • CMMS, FAULT / DRS, and local MSSQL islands already hold operational clues.
  • Robot, vision, and PdM information exists — but not in one reusable model.
Management message: preserve current investments and avoid operational disruption.

Gaps: why the current state does not scale

The plant cannot act like a modern maintenance intelligence environment until it solves the missing-middle problem: operational context, reusable event records, and a single user experience.

  • Different systems describe the same asset or event in different ways.
  • Maintenance teams still hunt across systems to diagnose and close issues.
  • Cloud analytics and AI do not receive enough clean data-in-context to create confidence.
Management message: the main problem is not another dashboard — it is missing context and interoperability.

To-Be: add the missing operating layer

The recommended future state keeps the current control stack in place and adds the layers that create operational clarity, better data quality, and enterprise reuse.

  • Historian for high-speed OT history and alarm/event storage.
  • An i3X-aligned context / UNS layer for asset modeling and event normalization.
  • One maintenance cockpit for faults, RCA, parts, SOPs, and escalation.
  • CESMII SM Profiles + i3X as the centerpoint for interoperable data-in-context.
Management message: this is how Nissan gets one asset model, one event record, and one escalation path.

Optimization Layer (future): better decisions on clean data

Once the data layer is live — historian feeding a cloud data platform with clean, contextualized asset and event records — three high-value optimization problems become tractable that classical schedulers handle poorly at Smyrna's scale. These are assignment problems, not prediction problems — the kind standard optimization tooling already solves well once the data underneath is clean and contextualized.

Maintenance scheduling

Given N assets, M maintenance windows, K technicians, parts availability, and production schedule constraints — find the optimal PM assignment. The paint shop is the highest-value target: scheduling PM there requires balancing multiple interdependencies simultaneously. Classical schedulers find local optima. This finds the global one.

Spare parts inventory allocation

Given failure probability distributions from the historian, parts lead times, and storage locations — find the optimal safety stock per location. Currently solved with gut feel or simple reorder rules. With clean historian data, failure rates become calculable. The allocation problem is a stochastic knapsack — a well-understood class of optimization problem.

Production resequencing

When an asset goes down mid-shift, resequencing production across downstream stations to minimize throughput loss is a real-time assignment problem. At Smyrna's line complexity this is computationally hard. The optimizer produces a resequencing plan in seconds so the Control Room acts on math, not instinct.

Built for speed, not a research project

The hard part is the clean, contextualized data the earlier phases produce — the optimization on top is standard tooling. Rationalizing asset and event models across lines and plants, once a slog, is now largely AI-assisted and done iteratively — closer to well-practiced open-source development than a multi-year program. The same clean foundation is what later makes capabilities like a live digital twin or fleet-wide learning practical — earned, not bolted on.

Key positioning: every dollar invested in historian quality and contextualization (Phases 1–4) directly improves optimizer output. The optimization layer is the return on that investment — available the day the data is clean enough to feed it. No other integrator in this space is formulating this today.
Detailed architecture

How floor data actually leaves the controllers — and where it gets stuck

The staged graphic shows the connectivity tier the As-Is story skips over. Here is the source-grounded detail behind it: the two connectors Nissan needs, the network zones they live in, what each source system can and cannot reach today, and the latency reality that makes "predictive" credible or not. Drafted from engineering working notes — to be validated with Nissan controls / IT.

Connectivity / protocol layer — two connectors, not one

FT Gateway — the Rockwell path

Native connector for the Allen-Bradley / Rockwell estate. At Smyrna it is deployed as numerous per-cell instances (each ABB cell has its own; =5 CIMPLICITY projects per instance), so per-instance throughput is not a concern.

Kepware — the multi-vendor path already owned

Covers GE Emerson PACSystems, Mitsubishi, Fanuc Focas, Modbus, and ABB that FT Gateway does not reach natively — and can publish directly to an MQTT / Sparkplug broker.

Together, not instead of each other: FT Gateway stays as the efficient Rockwell connector; Kepware becomes the normalization + publishing layer for everything else. They overlap only in the "act as an OPC server" role — they are complementary, not redundant.

Minimal new infrastructure to stand up: both connectors are already owned and proven in the plant, so the build adds a data and context layer — not a new gateway platform, operating system, and server estate to procure, secure, and support.

Management message: Nissan does not need a single universal gateway. It needs the right connector per protocol family — and it already owns the multi-vendor one.

Network zones

  • Plant network 4.3x.x.x — CIMPLICITY, database access, plant-level traffic.
  • Cell I/O network 4.33.x.x — control devices, localized per cell.
  • Safety P2P network 4.33.199.x — PLC peer-to-peer safety signal exchange.
  • DMZ — consolidated logging server (paint-db); single landing point today — a single point of failure and a segmentation consideration for scale.
  • IS / enterprise — replicated CMMS and FAULT data on MSSQL.

Robots sit on both the plant and I/O networks; each also carries an I/O adapter on the cell network. Asset identity is encoded in the third IP octet (cell number) — see the Data Standardization tab.

Management message: the zone separation is sound; the gap is that most floor context never crosses the DMZ to where analytics live.

Latency tiers — why "predictive" is not yet credible

Tier 0 — live
Live control / alarm (CIMPLICITY on floor): sub-second.
Tier 1 — historized
Historized OT (Proficy Historian target): seconds.
Tier 2 — replica
Cross-DMZ replica to IS: batch / periodic.
Tier 3 — PdM API
Senseye PdM API: hours.

A live command center needs Tier 0–1. Today much of the valuable signal sits at Tier 2–3 — which is why "predictive" cannot be credible until the data path is rebuilt.

Cell 022 (Paint Line 1, Sealer Automation) — pilot reality why the pilot cell is one of the harder cases
  • ABB robots, not Fanuc. RobView (not Fanuc Utilities) is the robot tool here — but it is connected to only 2 of 8 robots; the other 6 are visible through CIMPLICITY alarms only.
  • No RobView?SQL logging in 022 (only spray-booth cells do that); the sealer cell's robot history comes from CIMPLICITY SQL alarm logging.
  • Three ISRA vision systems, each logging to local MSSQL, deliberately kept on the cell I/O network and off the plant network to stop Windows updates from breaking the fragile vision software. Body-relevant data is dropped to the PLC. A replication mechanism does not exist yet and must be designed.
  • Elsewhere, Fanuc Utilities is antiquated and flagged for replacement.
Management message: the pilot cell is one of the harder connectivity cases, not the easiest — which makes it a strong proof point if the layer is built right.

Data standardization

From a raw PLC tag to a portable, reusable event object

Smyrna already uses consistent fault-tag and IP conventions — they just are not modeled into reusable context yet. Standardization is not a from-scratch exercise; it is capturing what exists and giving it one shared grammar. The clearest illustration is a single fault tag, named once, landed once, with a history and an escalation path.

The existing fault taxonomy (the raw input)

  • Conveyor faults: A.{project}.FLTxxxx  e.g. A.SLR_105.FLT0001
  • Cell rollup faults: PLC_CELL_MAJORFLT_xx_xx / PLC_CELL_MINORFLT_xx_xx  (the xx_xx is fault-word and bit)
  • Robot-specific faults: PLC_SLxxxx_MAJORFLT_xx_xx / _MINORFLT_xx_xx
  • Rollup: all CIMPLICITY projects roll up to a single plant FAULT project.

Worked example — raw tag to portable event object

PLC_SL2202_MAJORFLT_02_05  becomes a structured fault_event:

asset:
  site:        Nissan Smyrna
  area:        Paint
  line:        Paint Line 1
  cell:        022 (Sealer Automation)
  asset:       Robot SL2202 (ABB)
  network_id:  4.3x.22.202        # masked on public site
event:
  type:        fault
  severity:    major
  source_tag:  PLC_SL2202_MAJORFLT_02_05
  fault_word:  02
  fault_bit:   05
  source_sys:  CIMPLICITY (plant FAULT rollup)
  timestamp:   <iso8601>
action:
  owner:        <role>
  reason_code:  <std code>
  resolution:   <fix / part>
  verification: <closure evidence>

Same fault — now named once, landed once, with a history and an escalation path. The SM Profile / i3X promise made concrete.

Asset identity convention — already in use the IP scheme is a head start on the naming standard

Asset identity is encoded in the IP scheme: the third octet is the cell number, and the device number carries through both networks. A robot in cell 022 appears as 4.3x.22.2nn on the plant net and 4.33.22.2nn on the I/O net; the cell PLC carries the same cell octet. This existing convention is a head start on a formal asset-naming standard — capture it, do not reinvent it.

The four standardization deliverables

Asset hierarchy

site → area → line → cell → asset → subasset. Anchored on the IP-octet convention already in use.

Event dictionary

fault / alarm / downtime / work-order / quality / robot / vision / PdM — one record shape per event class.

Action taxonomy

owner / reason / closure / evidence — the minimum fields that make a fault closeable and auditable.

SM Profile + i3X workbook

the mapping that lets the model travel across systems instead of being welded to one product.

Nissan-friendly translation: one naming scheme, one event contract, one action record — built on conventions Smyrna already follows, so the cockpit and AI can finally trust the data.
Compute placement

Process where it’s right — edge and cloud by design

The question is no longer cloud versus on-prem. It is where each job belongs. Modern open models are small and capable enough to run real inference at the edge — so latency-critical and high-volume work stays on the floor, and only the work that genuinely benefits from scale goes to the cloud. Sending everything upstream is how cloud bills spiral: you pay to move and reprocess data that never needed to leave the plant.

Runs at the edge
  • Tier 0–1 latency — live control, alarms, guided triage, and anomaly checks that must answer in sub-second to seconds.
  • High-volume, routine inference — condition monitoring and vision on continuous streams, where per-event cloud cost would dwarf the value.
  • Data that should stay local — floor signals kept on the plant network by default; only curated results move up.
Belongs in the cloud
  • Fleet and cross-site learning — patterns that only emerge across lines, plants, or the Americas.
  • Heavy training and back-testing — model building on long history, run occasionally, not per event.
  • Enterprise analytics and reporting — where curated, contextualized events land for the wider business.
Why it matters: placing compute deliberately keeps troubleshooting fast, keeps sensitive floor data local, and keeps cloud spend tied to work that actually needs the cloud — not a bill for reprocessing everything twice.

Why CESMII matters here

CESMII i3X is not a side note — it is the organizing principle

Use SM Profiles as the common semantic model and i3X as the open interaction pattern. That keeps Nissan from hard-wiring plant context to one application and makes the future architecture more portable. These are published, industry-governed models — an open standard the sector is converging on, not a proprietary model authored and maintained by the firm that built it.

Translated for Nissan management

  • One naming scheme across lines, cells, robots, and utilities.
  • One event contract across faults, downtime, work orders, and predictive alerts.
  • One shared context model that can feed the cockpit, the Control Room, and cloud analytics.
  • Governance is alignment and leverage, not a committee. Ownership and lineage live in the model itself — teams agree on the shape once, then extend it iteratively, the way well-run open-source projects do.
  • New assets, lines, and other plants fold in by reusing the model, not by re-litigating it — governance you can see, applied at the speed of the work.
