Architecture story

As-Is: the plant already has useful systems

Nissan does not need a “rip and replace” story. The current stack already contains valuable signals and work processes. The challenge is not lack of systems; it is fragmentation.

  • CIMPLICITY remains the live control and HMI backbone.
  • CMMS, FAULT / DRS, and local MSSQL islands already hold operational clues.
  • Robot, vision, and PdM information exists — but not in one reusable model.
Management message: preserve current investments and avoid operational disruption.

Gaps: why the current state does not scale

The plant cannot act like a modern maintenance intelligence environment until it solves the missing-middle problem: operational context, reusable event records, and a single user experience.

  • Different systems describe the same asset or event in different ways.
  • Maintenance teams still hunt across systems to diagnose and close issues.
  • Cloud analytics and AI do not receive enough clean data-in-context to create confidence.
Management message: the main problem is not another dashboard — it is missing context and interoperability.

To-Be: add the missing operating layer

The recommended future state keeps the current control stack in place and adds the layers that create operational clarity, better data quality, and enterprise reuse.

  • Historian for high-speed OT history and alarm/event storage.
  • HighByte or equivalent context layer for asset modeling and event normalization.
  • One maintenance cockpit for faults, RCA, parts, SOPs, and escalation.
  • CESMII SM Profiles + i3X as the centerpoint for interoperable data-in-context.
Management message: this is how Nissan gets one asset model, one event record, and one escalation path.

Detailed Architecture: follow-on work to complete

This tab intentionally marks the work still needed before build. It gives Nissan a clear place to capture source-by-source design detail once owners, access paths, latency needs, and security rules are confirmed.

  • Validate each source: CIMPLICITY, CMMS / Maximo, FAULT, DRS, Senseye, RobView, ISRA vision PCs, drawings, and SOP repositories.
  • Define the OT / DMZ / IS data path, including what is read-only, what can be written back, and what must remain isolated.
  • Document latency targets: live alarms, event history, work orders, predictive exceptions, and cloud publishing cadence.
  • Specify integration ownership, support model, cybersecurity constraints, and fallback behavior for the paint-shop pilot.
Placeholder deliverable: detailed interface map, data-flow diagram, read/write matrix, ownership matrix, and pilot architecture for Cell 022 / paint shop.

Data Standardization: follow-on work to complete

This tab marks the standards work needed to make the architecture useful. The goal is not just moving data; it is moving trusted, reusable data-in-context.

  • Define asset hierarchy: site, area, line, cell, asset, subasset, robot, vision unit, tool, and utility asset.
  • Define event model: fault event, alarm event, downtime event, work order, maintenance action, quality event, robot event, vision event, and PdM signal.
  • Standardize reason codes, symptom codes, closure codes, recurrence flags, ownership fields, and required evidence.
  • Map the plant model to CESMII SM Profiles and i3X-style interfaces so the model can travel across systems.
Placeholder deliverable: Nissan Smyrna event dictionary, asset naming standard, minimum closure-data rules, data-quality checks, and i3X mapping workbook.

Optimization Layer: quantum-ready decisions on top of clean data

Once the data layer is live — historian feeding Databricks or Snowflake with clean, contextualized asset and event records — three high-value optimization problems become tractable that classical schedulers handle poorly at Smyrna's scale. These are combinatorial assignment problems, not prediction problems. The formulation runs today on classical hardware with a drop-in path to quantum annealing when problem scale warrants it.

Maintenance scheduling

Given N assets, M maintenance windows, K technicians, parts availability, and production schedule constraints — find the optimal PM assignment. The paint shop is the highest-value target: scheduling PM there requires balancing multiple interdependencies simultaneously. Classical schedulers find local optima. This finds the global one.

Spare parts inventory allocation

Given failure probability distributions from the historian, parts lead times, and storage locations — find the optimal safety stock per location. Currently solved with gut feel or simple reorder rules. With clean historian data, failure rates become calculable. The allocation problem is a stochastic knapsack — the exact class quantum annealing solves natively.

Production resequencing

When an asset goes down mid-shift, resequencing production across downstream stations to minimize throughput loss is a real-time assignment problem. At Smyrna's line complexity this is computationally hard. The optimizer produces a resequencing plan in milliseconds so the Control Room acts on math, not instinct.

Technology path

Simulated annealing (classical) runs today — no special hardware, ~5ms per solve. D-Wave Advantage2 is available via cloud API for problems exceeding ~50 assets with full constraint sets. Gate-based quantum (IBM, IonQ) unlocks additional problem classes in 5–10 years. The formulation built today runs unchanged on future hardware.

Key positioning: every dollar invested in historian quality and contextualization (Phases 1–4) directly improves optimizer output. The optimization layer is the return on that investment — available the day the data is clean enough to feed it. No other integrator in this space is formulating this today.

Why CESMII matters here

Use SM Profiles as the common semantic model and i3X as the open interaction pattern. That keeps Nissan from hard-wiring plant context to one application and makes the future architecture more portable.
