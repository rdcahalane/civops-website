Maintenance Systems Recommendation

This working recommendation focuses on what to keep, what to add, and how to get to a true single pane of glass without forcing maintenance, operators, and the Control Center into multiple disconnected tools.
In one sentence: keep the current control stack, add a purpose-built data and context layer, give people one maintenance cockpit, and make the plant feel like a 24/7 remote operations center with maintenance intelligence.
1. One cockpitOperators, maintenance, Control Room, and experts all work from the same event record.
2. Better dataHistorian + HighByte turn local islands into usable operational context.
3. Future-readyCurated data flows to Databricks / Snowflake for analytics, AI, and fleet learning.

Executive Recommendation

Recommended path: keep CIMPLICITY in place, add Historian + HighByte, use one frontline maintenance cockpit, and publish curated data to Databricks or Snowflake. If Nissan wants a broader human-performance layer, shortlist Murray Mentor, DeepHow, or Dozuki for the knowledge / training gap.

If Nissan wants the best practical architecture, keep CIMPLICITY as the live control layer, add a historian if they want an efficient operational record beyond SQL, add a contextualization layer such as HighByte, and present everything through one maintenance/operations interface.

Best-fit pattern: control at the edge, context in the middle, single pane for users, cloud lakehouse for analytics and AI.

Recommended stack by layer

Plant Control CIMPLICITY remains the HMI/SCADA backbone for live plant operations.
Operational History Add Proficy Historian if they want fast, efficient time-series and alarm/event storage beyond SQL.
Context / UNS HighByte standardizes assets, tags, and event models so data becomes reusable across systems.
Frontline UI Operations Hub, Workflow, Tulip, or a custom app can give users one screen for faults, RCA, parts, and escalation.
Cloud Analytics Databricks or Snowflake can serve as the enterprise analytics / AI layer once the data is contextualized.
Standards CESMII SM Profiles and i3X are the interoperability contract worth aligning to.

Why historian is worth paying for

  • SQL is fine for records, downtime logs, and work orders, but it is not the best long-term home for dense machine signals.
  • Historian is purpose-built for high-speed time-series and alarm/event data.
  • It gives faster retrieval, better compression, and cleaner trending for bad-actor analysis and AI feature generation.
  • It lets SQL stay focused on transactional data instead of becoming the catch-all for everything.

Best option paths

  • Proficy-first: CIMPLICITY + Historian + Operations Hub + Workflow + Plant Applications.
  • Hybrid: CIMPLICITY + HighByte + cloud lakehouse + single front-end app.
  • Open app stack: Ignition + HighByte + historian or SQL + cloud lakehouse.

Key Talking Points

What stays

  • CIMPLICITY as the plant control backbone.
  • SQL for transactional and reference data.
  • Existing local systems where they are already stable and fit for purpose.

What gets added

  • Historian for operational data retention and fast retrieval.
  • HighByte or equivalent for normalization, asset modeling, and UNS.
  • A single maintenance cockpit for alarms, faults, guidance, parts, and escalation.
  • Databricks or Snowflake for enterprise analytics and AI.

What CESMII / i3X means here

CESMII SM Profiles define the semantic model. i3X defines the open API style for interacting with those contextualized manufacturing objects. They are not the full solution, but they are the interoperability layer that makes the rest of the stack portable.

For Nissan, treat SM Profiles as the common language and i3X as the open interface strategy. That is especially useful if they want to mix Proficy, HighByte, Ignition, and cloud platforms without hard-locking into one vendor.

Ideal Maintenance Troubleshooting UX

The user experience should behave like a decision tree that starts simple and only adds complexity when needed. The operator should not have to know which system owns which data. The interface should guide the next best step.

1. Operator resolves independently The UI shows the likely issue, a safe first action, a short SOP, and a confirmation step. The operator can clear a minor fault, document the action, and continue without opening a maintenance ticket.
2. Operator requests help If the operator cannot clear the issue, the UI captures a structured fault report, starts the downtime clock, attaches context automatically, and alerts frontline maintenance with the relevant asset, symptom, and history.
3. Frontline maintenance resolves it Maintenance sees live alarms, similar failures, parts availability, drawings, procedures, and the most likely root cause. They can resolve, annotate, and close with standardized codes without needing L2 intervention.
4. L2 / Control Room / remote expert escalation If the issue is outside frontline scope, the UI creates a collaborative escalation case with live notes, photos, event history, and ownership. The remote expert should see the same context the tech sees, not a stripped-down ticket.

Recommended state model for the interface

Green Normal operation, no action needed.
Amber Developing issue or degraded condition, but not yet a stop.
Red Active downtime or fault requiring action.
Blue Predictive alert, recommended inspection, or maintenance window action.

What the screen should show at every stage

  • Top: plant, line, asset, shift, and event status.
  • Center: current condition, recommended action, and fault workflow.
  • Right: related incidents, manuals, PM history, parts, and expert chat.
  • Bottom: event timeline, handoff notes, and closure / follow-up actions.

Predictive / pre-fault experience

When there is no actual stop, the user interface should still surface the issue as a developing condition. This lets operators and maintenance act before the asset fails. The best pattern is a “watch list” or “degrading asset” lane that ranks issues by risk, confidence, and time-to-failure estimate.

Watch Potential issue detected. Monitor and log if it worsens.
Inspect Action recommended during the next available window.
Intervene High-confidence developing failure requiring immediate response.
The key UX principle is progressive disclosure: show the simplest useful action first, then expand context only when the user needs help.

Quantum-Ready Optimization

Once the data layer is live — historian feeding Databricks or Snowflake with clean, contextualized asset and event records — three high-value optimization problems become solvable that classical schedulers handle poorly at Smyrna's scale. These are not AI prediction problems. They are combinatorial decision problems: given a set of constraints, find the optimal assignment. That class of problem maps directly to quantum computing, and the formulation can run today on classical hardware with a drop-in quantum path when the problem scale warrants it.

Maintenance scheduling Given N assets, M maintenance windows, K technicians, parts availability, and production schedule constraints — find the optimal preventive maintenance assignment. The paint shop is the highest-value target: paint line downtime is disproportionately expensive and scheduling PM there requires balancing multiple interdependencies simultaneously. Classical schedulers find local optima. This finds the global one.
Spare parts inventory Given failure probability distributions from the historian, parts lead times, and storage locations across the plant — find the optimal safety stock allocation per location. Currently solved with gut feel or simple reorder rules. With clean historian data, failure rates become calculable. The allocation problem is a stochastic knapsack — the exact problem class quantum annealing solves natively.
Production resequencing When an asset goes down mid-shift, resequencing production across downstream stations to minimize throughput loss is a real-time assignment problem. At Smyrna's line complexity, this is computationally hard. The optimizer produces a resequencing plan in milliseconds so the Control Room acts on math, not instinct.

How it works with the stack being built

Data feeds in Historian + CMMS provide asset health scores, PM history, failure rates, and parts usage — the optimizer's inputs.
Constraints are encoded Production schedule, crew availability, budget, and downtime risk are expressed as a quadratic constraint problem (QUBO).
Optimizer runs Simulated annealing today (~5ms, runs anywhere). D-Wave quantum annealing when asset count exceeds ~50 with full constraint sets.
Plan surfaces in cockpit The optimal maintenance schedule and resequencing plan appear in the same frontline UI — no separate tool, no re-entry.

Why this is different from predictive maintenance AI

Predictive maintenance (what you already get from Databricks) Tells you which asset is likely to fail and when. Output is a probability and a recommended inspection window. Every major vendor sells this. It is a necessary capability but not a differentiator.
Quantum-ready optimization (what this adds) Takes the predictive output as an input and answers: given everything we know about failure risk, crew, parts, and production schedule, what is the mathematically optimal plan? No other integrator in this space is formulating this today.
Positioning for Nissan: the data layer being proposed in phases 1–4 is exactly the prerequisite this needs. Every dollar invested in historian quality and contextualization directly improves the optimizer's output. The optimization layer is the return on that investment — and it is available the day the data is clean enough to feed it.

Quantum technology status — honest framing

Today (classical SA) Simulated annealing solves the same QUBO formulation on standard hardware. Runs in milliseconds for ≤50 assets. Available now, no special infrastructure, no vendor lock-in.
Near-term (D-Wave cloud) D-Wave Advantage2 handles ~5,000 binary variables natively. For Smyrna's full asset network with real constraints, this is where classical SA starts to find local optima and quantum annealing finds better solutions. D-Wave Leap API available today via cloud — no hardware on-site required.
Future (fault-tolerant QPU) Gate-based quantum computers (IBM, IonQ) will unlock additional problem classes — multi-period scheduling, stochastic supply chain, and fleet-level optimization across multiple Nissan plants. Expected meaningful capability: 5–10 years. The formulation built today runs unchanged on future hardware.

Alternatives

  • Stay Proficy-centric: best if they want minimum disruption and the cleanest path inside the existing ecosystem.
  • Introduce Ignition: best if they want a flexible application layer and broader openness at the edge.
  • Introduce HighByte: best if the main pain is inconsistent naming, siloed MSSQL stores, and AI-readiness.
  • Go cloud-first analytics: best if the goal is to make plant data usable in enterprise AI and reporting quickly.

Bottom Line

The best recommendation is not “replace everything.” It is to create a layered architecture: keep the good control systems, add a historian where the OT data volume warrants it, normalize the data with HighByte or similar, give users one operational interface, and send curated data to Databricks or Snowflake for AI.

That gives Nissan a stronger near-term maintenance experience and a better long-term data foundation.
