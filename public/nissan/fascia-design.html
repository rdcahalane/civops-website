Fascia & Plastics — Design Detail

Architecture consistent with the maintenance project

The Fascia engagement uses the same underlying architecture assumptions as the maintenance modernization work — Cimplicity as the central system, a context and data layer on top, and a single operating experience for the people who work in the area.

Design principle
Retain and extend — don't replace. The same principle that governs the maintenance architecture applies here: build on FAULT, build on Cimplicity, and design the data layer so it feeds both Fascia operations and enterprise analytics without a redesign later.
CimplicityCentral system candidate — consistent with plant-floor architecture on the maintenance side.
FAULT spineAll help-call and downtime data integrates into FAULT — not alongside it.
Fabric pathSpecification written with Microsoft Fabric in mind — no redesign needed for the digital twin phase.
Technical foundation

Four architecture components

The functional specification will confirm which components apply and in what order. Each is consistent with the architecture already in use on the maintenance side of the project.

Cimplicity as the central system

Consistent with the plant-floor architecture used on the maintenance side. Retain and extend rather than replace — the same principle applies in Fascia.

Integration scope confirmed during connectivity feasibility workstream.

FAULT as the data spine

Nissan's standardized downtime tracking system. All help-call, downtime, and root-cause work integrates into FAULT — not alongside it. Reporting improvements target the DRS output layer.

Japan-format reporting gap addressed without touching FAULT's data model.

Historian + context layer

High-speed time-series and event data infrastructure — same approach as the maintenance project. Connects Cimplicity signals to a reusable operational data layer that feeds both the Fascia cockpit and enterprise analytics.

Reuse of the same data backbone reduces integration cost on follow-on phases.

Microsoft Fabric path

Craig's identified long-term direction. The functional specification will document the Fabric integration point so the follow-on implementation can reach it without a redesign — Brooksource's Microsoft partnership is available as a resource.

Digital twin capability becomes a natural extension, not a separate program.
Key principle: bring in data and systems engineering contacts early — before scope is finalized — to avoid the delays experienced on the control room project. Timing matters.
Deliverables

What this engagement produces

Six concrete deliverables — each designed to stand alone and to feed directly into implementation. No discovery will need to be repeated when the build phase begins.

Deliverable 1
Current-state visibility map Systems, data owners, naming conventions, FAULT data flows, and the DRS-to-Japan reporting gap — documented and validated with the Nissan team.
Deliverable 2
OEE requirements & methodology Accurate OEE definition including quality losses, hour-by-hour throughput logic, and alignment with Japan reporting expectations.
Deliverable 3
Connectivity feasibility matrix Equipment-by-equipment assessment of what can be connected, at what cost, and in what timeframe — scoped to injection molding, welder PCS, and punch machines.
Deliverable 4
Help-call workflow requirements Structured help-call and downtime workflow spec — routing, escalation, FAULT integration, and close-out logic — designed to drive adoption without mandating behavior change.
Deliverable 5
Quality KB requirements Specification for a centralized quality knowledge base: event classification, root-cause taxonomy, resolution pattern capture, and search requirements.
Deliverable 6
Functional specification + implementation estimate Build-ready functional spec covering all workstreams, data flows, integration points, and a follow-on implementation estimate scoped to the Fascia area.
