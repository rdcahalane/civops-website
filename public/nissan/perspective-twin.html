Overview / Live proof / Perspective twin

The same Cell 056 screen — in Ignition Perspective

The maintenance cockpit is just one consumer of the Unified Namespace. To prove the value layer isn't locked to a single tool, here is the Cell 056 Asset-Health screen rebuilt in Ignition Perspective's component idiom, bound to the identical namespace topics (mHv1.0/axiom/nissan-ccr) and reading the same live UNS bridge the web cockpit reads. Swap the rendering tool; the data, the model, and the topics don't change.

What this is, precisely. This is a Perspective-idiom rendering that consumes the live namespace through the bridge — the portability proof. A native Perspective view runs inside the Ignition gateway that already sits on this stack (v8.3.7, Perspective module loaded); wiring it natively adds the MQTT Engine module (or an OPC binding) and a Designer pass. The point stands either way: the exact same UNS topics drive it.

Why a second rendering matters

The namespace is the product; the UI is a choice

  • The web cockpit and this Perspective view are two consumers of one namespace — neither owns the data.
  • Nissan can standardize on Ignition Perspective, Power BI, Grafana, or Humble/Eva without re-plumbing a single source.
  • The tag paths shown at right are the real UNS topics — the same ones HighByte and MaestroHub serve.

Going native inside the gateway

The Ignition gateway is already on the stack. To run this as a native Perspective session served by the gateway itself:

  • Add the MQTT Engine (Cirrus Link) module to subscribe to the broker directly, or bind through the existing OPC UA connection.
  • Import this view into a Designer project and publish — the component tree above is the layout it targets.

Either way, the binding is to the same mHv1.0/axiom/nissan-ccr topics.
