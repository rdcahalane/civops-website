The Cell 056 cockpit — a working app, not a picture

This is the tablet a maintenance technician actually holds, running off the Unified Namespace. It's a real web application — the tiles, health scores, OEE, and axis torque are populated by a live data layer that updates on screen. Today it runs on the same representative/live feed as the rest of the proof; pointing it at the production UNS bridge is a one-function swap, not a rebuild. The same app can be re-skinned in Ignition Perspective or Power BI against the identical topics — the cockpit is just another consumer of the namespace.

Why this is the working demo

A consumer of the namespace — swap the data source, not the app

How "live" actually works here

  • The screen is driven by a data layer (Feed), not baked-in numbers — watch the axis torque and OEE trend update on their own.
  • Going to production data is one function: Feed.connectUNS() subscribes to mHv1.0/axiom/nissan-ccr/# on the broker (via the hosting bridge) and the same render code draws it.
  • BC5602's axis torque is the leg already flowing live end-to-end (Ignition → HighByte → UNS) — shown here as the live-source example.
OEE & the Quality gap: Availability and Performance compute from data Nissan has provided (runtime/faults, cycle counts). Quality has no source feed yet — scrap/rework isn't in the extracts — so it's flagged est. and OEE is labeled accordingly. Naming that gap is the point: it's the one feed to wire for a true, defensible OEE.

Same cockpit, any tool

This web app is one rendering. The identical namespace topics drive a Humble/Eva view, an Ignition Perspective screen, or a Power BI board — proof the value layer isn't locked to one platform any more than the source is.
