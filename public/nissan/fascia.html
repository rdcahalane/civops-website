The challenge

What is holding Fascia back

Fascia and Plastics operate with capable equipment and a knowledgeable team — but the data infrastructure does not match the operational complexity. The result is reactive maintenance, unreliable OEE metrics, and a help-call process that depends on tribal knowledge and informal communication.

Problem 1

OEE numbers don't reflect reality

Current reported OEE of 98–99% excludes quality losses and relies on built-in buffers. The methodology needs to be defined accurately before any improvement program can be measured.

Problem 2

FAULT reporting doesn't meet Japan's format

FAULT is the standardized downtime system — but DRS reporting output doesn't match what Nissan Japan expects. The gap is in presentation and aggregation, not the data itself.

Problem 3

Cultural adoption falls short of data accuracy

Education alone hasn't closed the data entry gap. The team needs solutions that surface useful information even when individual entries are incomplete or inconsistent.

Problem 4

No real-time throughput visibility

Hour-by-hour production status across injection molding, welder PCS, and punch machines requires manual aggregation. There is no live view that connects equipment state to throughput.

Problem 5

Help-call system lacks technician pull

When there is no clear benefit for the person entering data, adoption suffers. A help-call system that prioritizes, routes, and closes the loop gives technicians a reason to use it.

Problem 6

Connectivity feasibility is unknown

Before any integration can be scoped, the team needs a clear picture of what the injection molding machines, welder PCS, and punch machines can actually expose — and at what cost.

Engagement structure

Eight workstreams, one functional specification

The engagement is structured as a functional specification effort — not an implementation. The deliverable is a build-ready spec that makes follow-on implementation estimable, low-risk, and informed by real current-state data from the Fascia floor.

1 — Kickoff & alignment

Confirm scope, stakeholders, access, and the priority order for which area to tackle first. Shawn, Craig, and Paige aligned on how this connects to the control room work.

Outcome: shared understanding of scope and sequencing before any discovery work begins.

2 — Current-state mapping

Document existing systems, data owners, naming conventions, and where FAULT data actually lives vs. where it needs to go. Identify the entry points DRS uses and where the Japan format gap occurs.

Outcome: accurate as-is picture before requirements are written.

3 — Equipment connectivity feasibility

Assess what injection molding machines, welder PCS, and punch machines can expose — protocols, data types, polling rates, and integration cost. Map each machine to a feasibility rating.

Outcome: connectivity matrix that scopes the implementation realistically.

4 — OEE requirements

Define accurate OEE methodology: include quality losses, remove buffer padding, and establish hour-by-hour throughput as the core metric. Align with what Nissan Japan expects to see.

Outcome: OEE definition and measurement logic that reflects operational reality.

5 — Help-call & downtime workflows

Design a help-call system that technicians want to use: clear routing, escalation rules, priority logic, and close-out requirements. Integrate with FAULT rather than creating a parallel record.

Outcome: workflow specification that addresses data adoption at the source.

6 — Quality knowledge base

Define requirements for a centralized repository of quality events, root-cause classifications, and resolution patterns — so the institutional knowledge that currently lives in people's heads becomes searchable.

Outcome: KB requirements that survive personnel turnover.

7 — Traceability & automation opportunities

Identify where manual data collection can be reduced through automation — machine signals, barcode/scan events, or structured form capture — and document the traceability requirements for each.

Outcome: automation candidates with effort-to-value assessment.

8 — Functional specification package

Compile all workstream outputs into a build-ready functional spec: requirements, architecture, integration points, data flows, and a follow-on implementation estimate scoped to the Fascia area.

Outcome: a document that engineering can build from — no discovery re-work needed.

Technical foundation

Architecture consistent with the maintenance project

The Fascia engagement uses the same underlying architecture assumptions as the maintenance modernization work — Cimplicity as the central system, a context and data layer on top, and a single operating experience for the people who work in the area. The functional specification will confirm which components apply and in what order.

Cimplicity as the central system

Consistent with the plant-floor architecture used on the maintenance side. Retain and extend rather than replace — the same principle applies in Fascia.

Integration scope confirmed during connectivity feasibility workstream.

FAULT as the data spine

Nissan's standardized downtime tracking system. All help-call, downtime, and root-cause work integrates into FAULT — not alongside it. Reporting improvements target the DRS output layer.

Japan-format reporting gap addressed without touching FAULT's data model.

Historian + context layer

High-speed time-series and event data infrastructure — same approach as the maintenance project. Connects Cimplicity signals to a reusable operational data layer that feeds both the Fascia cockpit and enterprise analytics.

Reuse of the same data backbone reduces integration cost on follow-on phases.

Microsoft Fabric path

Craig's identified long-term direction. The functional specification will document the Fabric integration point so the follow-on implementation can reach it without a redesign — Brooksource's Microsoft partnership is available as a resource.

Digital twin capability becomes a natural extension, not a separate program.
Key principle: bring in data and systems engineering contacts early — before scope is finalized — to avoid the delays experienced on the control room project. Timing matters.
