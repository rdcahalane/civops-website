Kentucky Truck Plant — Approach

Real-time OEE visibility — across every dimension of KTP production

The goal is live, actionable visibility into everything that affects OEE at Kentucky Truck Plant: cycle time performance, uptime and downtime, and quality — drillable to a specific line, asset, shift, or vehicle configuration. That requires mapping what data exists, identifying the gaps, and connecting the sources that are already there.

Cycle time performance

The plant historian is the source of truth for equipment cycle signals. The functional spec will define which tags, at what polling rate, feed JPH and effective cycle time — by line, by area, and by vehicle configuration (Super Duty, Expedition, Navigator).

Quality yield

End-of-line and in-process inspection systems carry the quality event record. Connecting defects to the specific asset, shift, and vehicle configuration is what turns a count into a root-cause signal — and a driver of OEE, not just a separate metric.

OEE

OEE as the organizing framework

Every data source feeds one of the three OEE dimensions: the historian drives cycle time performance, the downtime system drives availability and cause, and quality inspection drives yield. The spec defines how they combine into an accurate, reportable OEE number — without buffers or manual adjustments.

Downtime & Help/Call

The downtime system carries the stop record. Help/Call carries response time and open ticket status. Both are operations metrics — visible alongside OEE so supervisors see cause and response in the same view, not in separate systems.

Drill-down to vehicle level

KTP produces multiple vehicle lines (F-250/350/450/550, Expedition, Navigator) with different cycle time targets and quality profiles. Visibility scoped to a specific model or configuration requires correlating equipment state with the vehicle sequence from MES.

What this is not: spare parts inventory, PM schedules, and long-range asset lifecycle planning are maintenance priorities — not the focus of this engagement. The focus is what affects throughput and quality today, visible in real time to the people who can act on it.
Engagement structure & timeline

Six phases → one functional specification

The engagement is structured as a functional specification effort — not an implementation. Six phases over approximately eight weeks, each ending with a Ford review checkpoint. The deliverable is a build-ready spec that makes follow-on implementation estimable, low-risk, and grounded in real current-state data from the KTP floor.

Phase 1 — 1 day

Kickoff & alignment

Confirm scope, stakeholders, system access, and the priority order for which production area or line to address first. Align on how this connects to any existing Ford digital manufacturing initiatives.

Outcome: shared understanding of scope and sequencing before any discovery work begins.
Phase 2 — 2 weeks

Current-state data mapping

Document existing systems — historian, downtime, quality inspection, MES — their data owners, naming conventions, and where each source lives vs. where it needs to go to support real-time OEE visibility.

Outcome: accurate as-is picture before requirements are written.
Phase 3 — 1 week

OEE methodology & gap analysis

Define accurate OEE methodology for KTP: cycle time by vehicle configuration, downtime attribution rules, quality loss inclusion. Identify the gaps between what's currently captured and what's needed for real-time visibility.

Outcome: OEE definition and measurement logic that reflects KTP's operational reality.
Phase 4 — 1 week

Help/Call & downtime workflow

Design a Help/Call system with automatic stop-state triggers where possible, clear escalation rules, and integration with the existing downtime system — so data lands once, in context, not twice in separate records.

Outcome: workflow specification that improves response time and data quality simultaneously.
Phase 5 — 1 week

Architecture & data translation layer

Define the architecture that connects historian, downtime, and quality data into a unified real-time view — including the recommended data translation layer, integration points, and the namespace structure that makes it scalable across lines.

Outcome: architecture recommendation grounded in KTP's actual systems.
Phase 6 — 1.5 weeks

Functional specification

Compile all workstream outputs into a build-ready functional spec: data flows, integration points, cockpit requirements, architecture diagrams, and a follow-on implementation estimate scoped to KTP.

Outcome: a document that engineering can build from — no discovery re-work needed.
